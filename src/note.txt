Summary

When we create a Bthread object, we create a stack and a context object 
and pass in a function pointer.

what we do is we set the context.rsp objects attribute to the top of the stack 
that was created

ctx.rsp = top; //<---- Important  (from Bthread.hpp)

then what we do is we set:

ctx.rip = (reg_t)thread_trampoline; 

So, when the thread_trampoline funciton begins getting executed, it will
exist in the function frame we created.  (so we don't have to worry about the 
function we are running accidentally running from the main thread)


So it doesn't start running of course until the scheduler calls the context_switch 
function in which case it will do the following:

jmp [rsi + 7*offset]         ; rip is last in struct

This sets the PC to the value stored at rsi + 7*offset which is a pointer to the function
in the text segment of the code.

Now, what this code in the text segment will do is it will activate a function 
frame that grows downwards starting where the rsp register is set. 
So it will actually make a new function frame beggining on the stack we made.



problem: We are only setting the ctx.rsp once, so after we context switch agian, 
it is still pointing to the thread_trampoline function, so it will literally just
run the function again without saving the rip register of where it was when it was
running the start_func(). So the function will run forever and never finish.

    ; Save current instruction pointer (RIP)
    lea rax, [rip + 0f]          ; address of label 0 (next instruction)
    mov [rdi + 7*offset], rax

So now, we are saving the RIP so it will return back to where it was inside of the
start_func() funciton

so mov destination, thing 
thing can be 3 things
#1: constant --> 0x1 for example (just a value)
#2: register --> r12 for example
#3: memory --> in this case you need [address] mov des, [address or label]
